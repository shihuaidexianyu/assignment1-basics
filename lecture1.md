随着模型参数规模的急剧增大（从 7.6亿 到 1750亿），**前馈神经网络（FFN）** 逐渐取代注意力机制（Attention）成为绝对的算力消耗主力（计算占比从 44% 飙升至约 77%），而 **Attention** 的计算占比反而显著下降（跌至 17%）；这说明在超大模型训练中，单纯优化 Attention 带来的整体收益会被巨大的 FFN 开销稀释，**FFN 才是大模型真正的计算瓶颈**。

## 涌现

1. 性能的“顿悟式”飞跃（The "Hockey Stick" Curve）

这是涌现最直观的视觉表现。

- **潜伏期：** 在模型规模（参数量或计算量）达到某个临界点之前，模型在特定任务上的表现**极度平坦**，几乎徘徊在随机猜测（Random Guessing）的水平，没有任何进步迹象。
- **突变期：** 一旦跨过这个特定的临界值，模型的能力曲线会**突然近乎垂直地拉升**。它不是从 10分 涨到 20分，而是直接从 0分 跳变到 60分 甚至更高。

2. 小模型的“不可外推性”（Unpredictability）

你无法通过观察小模型的表现来预测大模型是否会拥有某项能力。

- 如果我们观察一个 10亿 参数的模型（表现为 0）和一个 50亿 参数的模型（表现仍为 0），**线性外推**的结论是“这个任务太难，模型学不会”。
- 但现实是，到了 100亿 参数时，能力可能突然出现。这种表现打破了传统的 scaling law（在简单任务上性能随规模平滑增长的规律）。

3. 训练损失与任务能力的“割裂”（Loss vs. Capability Divergence）

这是涌现现象中一个非常反直觉的观测结果：

- **微观层面：** 模型的**训练损失（Training Loss）** 是随着训练过程**平滑、连续下降**的（一直在变好）。
- **宏观层面：** 模型在下游复杂任务（如算术、逻辑推理）上的**准确率** 却是**阶跃式**变化的（长时间不变，然后突然变好）。
- **表现：** 这意味着模型在“预测下一个词更准”这一指标上的平滑进步，在很长一段时间内**完全无法转化**为解决具体问题的能力，直到某一刻突然转化成功。

4. 复杂任务的“全有或全无”（All-or-Nothing）

涌现通常只出现在**多步骤、高复杂度**的任务中（如思维链推理、代码生成、三位数乘法）。

- 在这些任务中，模型要么完全不会（输出胡言乱语），要么就能做对。
- 很少观察到“做对了一半”或者“逐渐接近正确答案”的中间状态。表现形式上呈现出一种极端的二元对立。

## tokenizer

**Tokenizer（分词器）** 是所有现代自然语言处理（NLP）和大型语言模型（LLM，如 GPT、Llama）的**第一道门槛**。

简单来说，它的作用是**将人类能读懂的“文本”，转换成机器能读懂的“数字列表”**。

### 1. 核心概念：为什么需要 Tokenizer？

计算机本质上是一个数学计算器，它无法直接处理 "Apple" 这个单词，它只能处理数字（向量/矩阵）。

Tokenizer 的工作就是建立一个映射关系（Vocabulary/词表）：

1. **输入**：`"I love AI"`
2. **切分**：`["I", "love", "AI"]`
3. **映射**：`[23, 456, 1209]` （这些数字就是 Input IDs）

之后，模型（Model）会拿着这些数字去查表（Embedding Table），将其转化为高维向量进行计算。

> **关键点**：Tokenizer 决定了模型“看”世界的颗粒度。

---

### 2. 三种主流的分词粒度

Tokenizer 如何把一句话切开？这是分词器设计的核心难题。主要经历了三个阶段的演变：

#### A. 词粒度 (Word-based)

最直观的方法，按空格或标点切分。

* **例子**：`"unhappily"`  `["unhappily"]`
* **优点**：保留了完整的语义。
* **缺点**：
  * **词表爆炸**：英语有数十万单词，如果都要存，词表会巨大。
  * **OOV 问题 (Out of Vocabulary)**：如果训练时见过 "cat"，没见过 "cats"，模型就会把 "cats" 标记为 `<UNK>` (Unknown)，导致信息丢失。


#### B. 字符粒度 (Character-based)

把每个字母或汉字当做一个 Token。

* **例子**：`"unhappily"`  `["u", "n", "h", "a", "p", "p", "i", "l", "y"]`
* **优点**：词表很小（英语只要26字母+符号），几乎不会有 OOV。
* **缺点**：
  * **序列过长**：一句话变得非常长，大大增加了模型的计算负担（注意：LLM 的上下文窗口是有限的）。
  * **语义缺失**：单个字母 `"h"` 通常没有独立含义，模型需要花费更多层数去学习组合意义。


#### C. 子词粒度 (Subword-based) —— **当前主流**

这是目前 GPT、BERT 等模型都在使用的方案。它在“词”和“字”之间寻找平衡：**常用词保持完整，生僻词拆分为词根。**

* **例子**：`"unhappily"`  `["un", "happi", "ly"]`
* **原理**：
  * `"un"` (前缀，表示否定)
  * `"happi"` (词根)
  * `"ly"` (后缀，表示副词)


* **优点**：既控制了词表大小（通常 3万~10万），又解决了生僻词问题（通过组合认识的词根来理解新词），且序列长度适中。

---

### 3. 主流的 Subword 算法

你可能会经常听到以下几个术语，它们都是实现“子词切分”的具体算法：

| 算法名称 | 简介 | 代表模型 |
| --- | --- | --- |
| **BPE (Byte Pair Encoding)** | 统计频次，不断合并出现频率最高的相邻字符对。 | **GPT-2, GPT-3, GPT-4, LLaMA** |
| **WordPiece** | 类似 BPE，但在合并时考虑了语言模型的概率提升，而非仅仅是频次。 | **BERT, DistilBERT** |
| **SentencePiece** | Google 推出的一个库，把句子视为原始字节流（Raw Stream），无需预先按空格分词，对中文/日文等不使用空格的语言非常友好。 | **T5, LLaMA (实现层面)** |

**GPT-4 的 BPE 特性：**
现在的 LLM 通常使用 **Byte-level BPE**。它们不是处理 Unicode 字符，而是处理 UTF-8 字节。这意味着它们可以处理任何语言，甚至是 Emoji 表情符号或代码中的特殊符号，而不会出现乱码。

---

### 4. 实际工作流程：从文本到 ID

当你在 Python 中调用 `tokenizer` 时，内部发生了这四步：

1. **Normalization (标准化)**：
   * 清理文本，例如去除多余空格，统一转小写（如果是 uncased 模型），处理 Unicode 规范化（NFC/NFD）。


2. **Pre-tokenization (预切分)**：

   * 根据空格或标点符号，把句子粗略切分成“单词块”。

   * *中文特殊性*：中文没有空格，通常需要专门的切分逻辑或直接使用 SentencePiece。


3. **Model Encoding (模型切分)**：

   * 应用 BPE 或 WordPiece 算法，把单词块切分成最终的 Token（子词）。

   * 例：`"Transforming"`  `["Transform", "ing"]`


4. **Mapping (映射)**：

   * 查阅词表（Vocabulary），将 Token 转换为 ID。

   * 例：`["Transform", "ing"]`  `[10938, 283]`


---

### 5. 常见误区与特别注意

1. **Tokenizer != Embedding**

   * **Tokenizer**：把文本变成整数 ID（比如 `50256`）。

   * **Embedding**：模型的第一层，把整数 ID 变成一个 1024 维的浮点数向量。


2. **Token 数量 != 单词数量**

   * 在英语中，1000 个单词大约等于 1300 个 Token（经验值：1 Token  0.75 单词）。

   * 在中文中，现在的模型（如 Qwen, Yi）优化较好，通常 1 个汉字对应 1~1.5 个 Token，但在老模型（如 GPT-3）中，一个汉字可能被拆成 2-3 个 Token，非常费钱且占用上下文窗口。


3. **特殊 Token**

   * 词表中通常包含一些功能性 Token，例如：

   * `[CLS]` / `<s>`：句子开始。

   * `[SEP]` / `</s>`：句子结束。

   * `[PAD]`：填充位（因为矩阵运算要求长度一致）。

   * `[MASK]`：用于掩码语言模型训练。




### 个人的问题

#### 命题一：分词粒度与语义表征的非对齐性 (Tokenization-Semantic Misalignment)

#### 1. 表述

**假设:** 设人类认知空间中的语义概念集合为 $\mathcal{C}$，自然语言的文本空间为 $\mathcal{S}$。定义分词函数 $T: \mathcal{S} \rightarrow \mathbb{Z}^n$，其中 $\mathbb{Z}^n$ 为离散的 Token ID 序列空间。若 $T$ 的切分边界与 $\mathcal{C}$ 中概念的形态学边界（Morphological Boundaries）存在显著的非同态性（Non-homomorphism），即 $\exists s \in \mathcal{S}$，使得 $s$ 对应的单一概念 $c \in \mathcal{C}$ 被切分为无独立语义的子序列 $\langle t_1, t_2, ..., t_k \rangle$。

**推论:** 这种切分会导致模型 $M$ 在潜在空间（Latent Space）构建概念表征时引入额外的计算开销（Computational Overhead），并导致对长程依赖（Long-term Dependency）和特定领域（如算术、逻辑）的理解产生偏差（Bias）。

#### 2. 解答

这一问题揭示了当前 LLM 架构中**“统计压缩”与“语义完整性”之间的根本矛盾**。

- 表征碎片化 (Representation Fragmentation)：

  在机械可解释性研究中，我们通常寻找对应特定概念的“特征方向”（Feature Direction）。如果分词器将一个概念（如 "unbelievable"）切分为 ["un", "believ", "able"]，模型的第一层 Embedding 无法直接获得“难以置信”这个完整语义。

  - **后果**：Transformer 的前几层（Layer 0-2）必须充当“软性分词器”，利用注意力机制（Attention Heads）将分散的 Token 聚合（Aggregate）成一个完整的语义单元。这浪费了模型的深度和参数容量。

- 归纳偏置错配 (Inductive Bias Mismatch)：

  人类阅读是基于“语素”（Morpheme）的，而 BPE 算法是基于“频率”（Frequency）的。

  - **算术灾难**：这是最典型的例子。对于十进制数 $1000$，BPE 可能切分为 `[10, 00]`，而 $1001$ 切分为 `[100, 1]`。这种切分破坏了数字的位值系统（Place-value System）。模型难以通过简单的线性变换学习 $1000 + 1 = 1001$，而必须记忆大量的非线性映射。这也是为什么 LLM 在做大数加减法时表现不佳的根本原因之一——它们在“看”数字的方式上是“失读症”患者。

#### 命题二：分词映射的双射性与可逆性 (Bijectivity and Reversibility of Tokenization)

#### 1. 表述

**假设:**  定义分词过程为函数 Encode $E: \mathcal{S} \rightarrow \mathcal{V}^*$（其中 $\mathcal{S}$ 为所有可能的字符串集合，$\mathcal{V}^*$ 为词表中 Token ID 的所有有限序列集合）。定义解码过程为函数 Decode $D: \mathcal{V}^* \rightarrow \mathcal{S}$。

性质： 分词系统应当构成一个双射（Bijection），即满足严格的可逆性：$$\forall s \in \mathcal{S}, \quad D(E(s)) \equiv s$$

这意味着文本空间 $\mathcal{S}$ 与 序列空间 $\mathcal{V}^*$ 是同构的（Isomorphic）。

#### 2. 解答

在数学理想状态下，这个定义是完美的。但在工程实践中，这个“双射”通常是**有损的（Lossy）**或**有条件的（Conditional）**。

- 规范化破坏了单射性 (Normalization breaks Injectivity)：

  实际上，Tokenizer 通常包含一个预处理步骤 $P$（Normalization），例如去除多余空格、Unicode 规范化（NFC/NFD）或转小写。

  $$E(s) = \text{Model}(P(s))$$

  若 $P(s_1) = P(s_2)$ 且 $s_1 \neq s_2$（例如 $s_1=$"Latte"，$s_2=$"latte" 在某些 uncased 模型中），则 $E(s_1) = E(s_2)$。此时，$D(E(s_1)) = s_2 \neq s_1$。

  **结论：** 它是满射（Surjective）但非单射（Injective）。这意味着你无法总是无损还原原始文本的格式信息。

- Byte-level BPE 的修正 (Restoring Bijectivity)：为了逼近你提到的“完美双射”，GPT-4 和 LLaMA 等现代模型采用了 Byte-level BPE。它们将 $\mathcal{S}$ 定义为 UTF-8 字节流 而非 Unicode 字符流。

  - 在此定义下，只要输入是合法的 UTF-8 字节序列，现在的 Tokenizer 确实极其接近双射函数。即使是未见过的字符（OOV），也会回退到字节编码，从而保证信息不丢失。

- 从解释性角度看“双射”的意义：如果 $E$ 是双射，我们可以安全地假设 Token 序列 $\mathbf{z}$ 包含了输入 $s$ 的全部信息（Total Information）。这对于研究 LLM 内部机制至关重要：如果我们发现模型表现出某种偏见，我们可以确定这种偏见不是因为信息在分词阶段丢失了（例如把生僻字变成了 <UNK>），而是模型在处理这些 Token 时产生的内部动态。
